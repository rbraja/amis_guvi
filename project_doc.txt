How to run and test manually
1. Install dependencies:
```bash
pip install -r requirements.txt
```
2. (Optional) Build the vector DB from CSV sources:
```bash
python -m src.rag_setup
```
3. Run the Streamlit app and open the RAG Assistant page:
```bash
streamlit run app.py
```
4. On the RAG Assistant page, enter a question and click `Search`.
   - If the vector DB is present the Chroma backend will be used; otherwise the TF-IDF fallback will return results.
   - Use `Rebuild Vector DB` to recreate the collection from CSV sources.

Problem Statement:
UrbanHive Ops is a Bengaluru-based PropTech and integrated facility-management company (founded 2016) that operates maintenance, helpdesk, and asset care for 120+ premium apartment communities across South India. Their internal platform AMIS powers work orders, technician dispatch, SLA tracking, and resident feedback for every site they manage.

In 2026, UrbanHive's leadership launched a Data & AI program to reduce SLA breaches and scale help desk support. The Guvi AIML capstone uses an anonymized operational data export from their highest-volume pilot property,  GreenPark Meridian Residency (15 towers, 2,400 apartments, ~6,500 residents), covering roughly 24 months of AMIS transactions, service visits, asset events, and policy documents.

Learners act as data science consultants delivering the same outcomes UrbanHive's internal AI team is building: clean warehouse tables, predictive models, NLP on service text, and a deployed agentic RAG assistant for staff.

UrbanHive Ops at GreenPark Meridian processes thousands of maintenance tickets per quarter. Today this leads to:
Delayed resolution with no data-driven priority assignment
Poor technician matching (skill vs complaint type)
No visibility into asset health until failure
No analysis of seasonal complaint spikes (monsoon leakage, summer HVAC load)
Weak resident feedback loops after ticket closure
Management cannot answer policy/SLA questions at scale without searching documents
UrbanHive has engaged learners to deliver an end-to-end data & AI solution that:
Cleans and integrates AMIS operational exports (staging raw/ → analytics-ready tables)
Delivers dashboards and predictive models aligned to VP Operations and Head of Maintenance KPIs
Applies NLP/DL on resident complaints and feedback captured in AMIS
Deploys an AI maintenance assistant (RAG) on UrbanHive society rules, technician repair notes, and OEM asset manuals

Approach:
Learners follow a phased methodology across 14 days:
Phase 1 — Data Understanding & Cleaning (Days 1–3)
Profile all tables in raw/ for duplicates, nulls, invalid formats
Fix emails, dates, negative resolution times, legacy category codes
Produce cleaned_by_you/ with a documented cleaning log (reproducible notebook)
Validate row counts against assignment targets
Phase 2 — Integration & EDA (Days 4–5)
Join hub table maintenance_requests to apartments, residents, technicians, building_assets, service_logs, resident_feedback
Analyze complaints by tower, floor, season, category
Engineer features: season_flag, complaint_hour, is_repeat_complaint, asset failure counts
Phase 3 — Machine Learning (Days 6–8)
Multi-class: predict priority from structured + text features (TF-IDF + tabular)
Regression: predict resolution_time_hours (log-transform, MAE/RMSE)
Recommendation: rank technicians using technician_history success by category
Handle class imbalance using SMOTE / class weights where needed
Phase 4 — Deep Learning & NLP (Days 8–10)
Fine-tune DistilBERT (or RoBERTa) or meta on feedback_text for sentiment_label
Train complaint category classifier on complaint_text
Compare DL vs ML baseline (accuracy, F1, training time)
Phase 5 — Agentic RAG & Deployment (Days 10–14)
Chunk and embed: society_rules, service_logs.repair_notes, asset_manuals
Build retrieval pipeline (ChromaDB + all-MiniLM-L6-v2) or any 
LangGraph / LangChain agent with citations and confidence threshold fallback
Deploy Streamlit app: analytics tab + model inference + RAG chat
Publish GitHub repo + data export zip

SQL Example  (SQLite / PostgreSQL)

The query below joins the four core tables to produce a closed-ticket view with tower, resident, and technician context. Use it as a starting point for Phase 2 EDA joins.
SELECT
    m.request_id,
    a.tower_id,
    a.floor_number,
    m.category,
    m.priority,
    m.status,
    r.resident_name,
    t.technician_name,
    t.experience_years
FROM maintenance_requests m
JOIN apartments a         ON m.apartment_id = a.apartment_id
JOIN residents r          ON m.resident_id  = r.resident_id
JOIN technicians t        ON m.assigned_technician_id = t.technician_id
WHERE m.status = 'Closed'
LIMIT 100;
Tip: Replace WHERE m.status = 'Closed' with any status filter (Open, In Progress, Escalated) to slice the dataset differently during EDA.
